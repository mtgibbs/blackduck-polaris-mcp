## Codebase Patterns

- **3-layer architecture**: MCP Tool → Service Layer → API Layer
  - Tools in src/mcp/tools/ define schemas and call services
  - Services in src/services/ contain business logic
  - API layer in src/api/ handles HTTP calls with vendor media types
- **Service imports**: Always import from services/index.ts, never from api/ directly
- **ProjectMapping structure**: Has `id`, `projectId`, `configurationId`, `btsProjectKey`, `btsIssueType` (default issue type)
- **Error patterns**: Return `{error: string}` for user-facing error messages with actionable guidance

**Learnings for future iterations:**
- **Validation timing is critical**: When you have auto-resolution logic, validate AFTER resolution, not before
  - Early validation can't check if resolution succeeded in providing required values
  - Post-resolution validation catches all failure modes (missing params + failed resolution)
- **Schema descriptions matter**: Tool parameter descriptions should reference the correct tools users need to call
  - Incorrect tool references (like get_external_projects vs get_external_issue_types) confuse LLMs and users
- **Error messages should be specific and actionable**: Instead of generic "missing parameter" errors, tell users exactly which tool to call
- **Comment your non-obvious logic**: The validation reordering needed a comment explaining WHY it happens after resolution
