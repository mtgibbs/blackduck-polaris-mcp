## Codebase Patterns

- **3-Layer Architecture**: MCP Tool → Service → API
- **API Module Pattern**: Use `getClient()` from client.ts, create endpoint functions with vendor media types
- **Service Module Pattern**: Wrap API functions with Options interfaces, following naming convention (e.g., GetXOptions, CreateXOptions)
- **Type Organization**: Add domain types to src/types/polaris.ts, keep organized by feature area
- **Vendor Media Types**: All Polaris API calls require specific Accept/Content-Type headers (e.g., `application/vnd.polaris.policies.issue-policy-1+json`)
- **Pagination**: Use `client.getAllOffset()` for offset-based pagination (max 100 per page)

**Learnings for future iterations:**
- All policy API endpoints use /api/policies/ base path
- Issue policies use filterGroups with rules array
- PR policies use rules directly (not wrapped in filterGroups)
- Policy assignments require filter parameter for GET operations
- Policy settings endpoint uses Content-Type: application/json (not vendor type)
- Test scheduling policies do not support DELETE operation
- Maximum 5 rules per policy, 5 fixByRule entries per rule
